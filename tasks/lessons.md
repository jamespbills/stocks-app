# Lessons

- **Ticker matching across the engine and the brain must use the FULL ticker (suffix preserved) — never a suffix-stripped base.** The Markdown Reviews gate first matched `view_play_universe` tickers to brain pages via `normaliseTickerToken` (strips the exchange suffix: `ABC.L`→`ABC`), which would silently conflate a UK listing (`ABC.L`) with a different US one (`ABC`) — a wrong match in a decision tool, the worst failure mode. Both systems actually store the full suffixed ticker (confirmed: `migrations/002_app_portfolio_settings.sql` seeds `app_portfolio` with `('MTL.L'),('POWL'),('SPSY.L'),('SFM'),('YELP')…` which join `dim_companies` on the full ticker; UK = `.L`, US = bare). Fix: a dedicated `tickerKey(t) = t.trim().toUpperCase()` (suffix preserved) for cross-system joins, distinct from `normaliseTickerToken` (suffix-stripped, kept only for brain-internal filename/wikilink resolution where a lossy base is intentional). Principle: a missed match (shows as orphan/coverage-gap, visible) is always safer than a wrong match (silent). The exchange suffix in the brain's `ticker` frontmatter is the market disambiguator — preserve it. See `src/modules/markdown-viewer/gate/build-gate-rows.ts`.

- **NULL `date_released` reports were never ingestable — the skip filter predates every script version, and the existing NULL-date rows came from the 2025-04-03 bulk migration.** Both `ingestion_2.py` and `ingest_xbrl_3.py` (and the retired `Old\ingestion.py`) always skipped blank-date rows; the 72 NULL-date reports in `fact_reports` all share `created_at = 2025-04-03 23:03:41` (the `populate_fact_reports` seed). Fixed 2026-06-09: blank dates now ingest as NULL (warning logged, counted as `rows_ingested_without_date_released`), and the duplicate-UPDATE path uses `COALESCE(%s, date_released)` so a blank cell never wipes a stored date. Semantics: NULL `date_released` = historical backfill — already excluded everywhere it matters (TA backtest cohort filters `IS NOT NULL`, chart overlay drops undated markers, `view_earnings_calendar` ranks NULLs last). `release_lag_days` is vestigial: nothing reads it (only the one-time migration proc references it), so leaving it NULL is free — do NOT fabricate estimated release dates to fill it; that would poison backtest entry timing.

- **A renderer component importing one light helper from a module that *also* imports a node-only lib drags the whole lib into the renderer bundle.** Markdown Reviews' `Reader.tsx` imported `deriveTitle` from `entry.ts`; `entry.ts` also imports `parseFrontmatter` (→ gray-matter, which uses `eval` in `engines.js`), so Vite pulled gray-matter into the *renderer* chunk (a +600 kB chunk + an `eval is strongly discouraged` warning printed under "building client environment"). ES module imports are file-granular, not symbol-granular — importing any export evaluates the module's top-level imports. Fix: split the pure, dependency-free helper into its own file (`title.ts`) and import *that* from the renderer; keep `frontmatter.ts`/`entry.ts`/`index-builder.ts` (gray-matter) for the main process only. **Detection:** watch `npm run build` for a gray-matter `eval` warning or an unexpected large renderer chunk — that's the tell the boundary leaked. Related: the two tsconfigs — `tsconfig.node.json` includes only `electron/**`, `tsconfig.web.json` typechecks all `src/**` under DOM (no node types) — so an `fs`-using file must live under `electron/` (the `scanBrain` walk), while its pure logic (`indexBrainFiles`, `isSafeRelPath`) lives under `src/` and is unit-tested with in-memory data.

- **gray-matter / js-yaml parse YAML frontmatter with the *default* schema, which coerces and is unsafe for a stock brain.** Two traps the Markdown Reviews parser hit: (1) an unquoted `last_updated: 2026-06-07` is parsed into a JS `Date` object (UTC midnight) — the same `string`-typed-but-`Date`-at-runtime lie as mysql2 DATE columns; (2) the YAML "Norway problem" turns bare `NO`/`yes`/`on`/`off` into booleans, which would silently corrupt a ticker or a `play_tag`. Fix: parse through `JSON_SCHEMA` — `matter(raw, { engines: { yaml: (s) => yaml.load(s, { schema: yaml.JSON_SCHEMA }) } })`. Strings stay strings, ints stay ints, no date/bool magic. Also: gray-matter *throws* on invalid YAML (it does not swallow), so wrap `matter()` in try/catch to keep a malformed file indexable (`malformed: true`) instead of aborting the whole rescan. Note an unterminated double-quote sometimes parses leniently across lines; an unbalanced flow collection (`[{...]`) is a reliable "definitely-invalid" fixture for the graceful-fallback test. See `src/modules/markdown-viewer/frontmatter.ts`.

- **mysql2 DATE columns return JS `Date` objects at runtime**, not strings — even though TypeScript types them as `string | null`. Any sort, comparison, or string operation on a date field must handle `instanceof Date` (e.g. coerce via `.getTime()`). Do not rely on `String(value).localeCompare()` for date sorting.

- **`ticker` is not a unique row identifier.** `view_watching` can return multiple rows for the same ticker (one ticker, multiple active reports). Never use `ticker` alone as a React `key`, selection state, or `findIndex` target. Always derive a composite `rowKey` — e.g. `` `${ticker}|${filing_identifier ?? date_released ?? ...}` ``.

- **`useState(prop)` only initialises on mount — don't mount with a placeholder, then update the prop.** If a component's local state is seeded from a prop (`useState(detail.comboNote)`), and you mount the component with an empty/placeholder version of that prop before async data arrives, the state will be stale for the component's lifetime (even after the prop updates in the parent). Fix: delay mounting the component until real data is available. A loading placeholder (rendered when `selectedCombo && !comboDetail`) is cleaner and safer than an immediately-mounted shell.

- **Never call `.toISOString()` on a mysql2 Date value for display or intermediate storage.** `.toISOString()` converts to UTC, which shifts the date one day back in BST (UTC+1). Use `formatDate(value, 'iso')` from `src/lib/format.ts` instead — it uses local date getters (`getFullYear`, `getMonth`, `getDate`) and is correct in all timezones.

- **react-hooks/set-state-in-effect (ESLint plugin v7.1.1) blocks synchronous `setState` calls in `useEffect` bodies.** Restructure fetches to use `.then()` callbacks, or ensure all state updates are inside the async callback, not synchronously before it. `set-state-in-effect` fires on any setState call that isn't inside an async callback or event handler inside the effect.

- **react-hooks/refs-during-render (same plugin) blocks reading `ref.current` during render.** It only allows ref reads inside event handlers, effects, or layout effects. When a child component needs a stable DOM element from a parent (e.g. anchoring a popover to its trigger), the cleanest fix is a ref **callback** that updates `useState`: `const [el, setEl] = useState<HTMLDivElement | null>(null); <div ref={setEl} />`. For dynamic per-key collections (e.g. one element per ticker on the calendar), keep a `useRef<Map>` for assignment, then sync the *active* element into state via a `useLayoutEffect` keyed on the active identifier. Never assign `ref.current = …` inside the function body.

- **Popovers whose anchor sits inside an `overflow: hidden` container must portal to `document.body`.** Absolute positioning relative to the anchor is silently clipped to nothing. Use `createPortal`, `position: fixed`, and compute `top`/`left` from `anchorEl.getBoundingClientRect()` in a `useLayoutEffect`. Always flip to the anchor's left side when the right placement would clip the viewport, and clamp the top so the popover stays on-screen near the bottom edge. Reposition on `scroll` (capture phase) and `resize`. See `src/components/Popover.tsx`.

- **Outside-click dismiss via a global `window` click listener must check `contains(e.target)`.** Bare `onClick={(e) => e.stopPropagation()}` on the popover root does NOT stop native bubbling to `window` in React 17+ (events are delegated at the root container, not document). Without the containment check, every click inside the popover would also dismiss it.

- **mysql2 DECIMAL columns return strings at runtime, not numbers.** Even though TypeScript types them as `number | null`, mysql2 sends DECIMAL values as strings (e.g. `"14.0000"`). Strict equality (`=== 13`) will never match. Always parse at the `rowToEntry()` boundary: `Math.round(Number(row.field))`. This applies to any numeric column typed DECIMAL in MySQL — play scores, percentages, ratios, etc.

- **When `set-state-in-effect` fires on a loading flag, use a derived variable instead of a state setter.** Rather than `setLoading(true)` synchronously at the top of a fetch effect (blocked by the rule), track *what data is currently shown* in a separate state value (e.g. `loadedStrategy: Strategy | null`) and derive loading: `const isLoading = loadedStrategy !== strategy`. Call `setLoadedStrategy(strategy)` inside `.then()` and `.catch()` where the rule permits setState. This also correctly shows the loading state on any data-source change (strategy switch, refetch) without extra state management.

- **For Disregard pre-fill on the Report Calendar, derive YEAR from the year part of the report's relevant date and FILING from `view_earnings_calendar.next_expected_filing`** — never from `r_financial_year` or `r_filing_identifier`, which reflect the company's currently-tagged FY and the *last* filed report respectively. The Disregard form is editable so this is a best-guess pre-fill, but defaulting to the next-upcoming report's year/filing is what the user expects.

- **Coverage status: a target window end in the future should be treated as "ongoing", not a gap.** When a play or tracked ticker's `target_to` date is in the future (because `last_release + trail_days` hasn't elapsed yet), comparing the archive's `last_held` against that future date will always show a "tail gap" — but the data literally can't exist yet. The fix: in `deriveStatus()`, treat `targetTo > todayIso` as `ongoing = true` and cap `effectiveTo` at today. Only then does "covered to today" correctly mean `fresh`. Applies to any archive-coverage status function.

- **Migrations re-run on every boot — they must be fully idempotent.** The `runMigrations()` runner in `electron/migrations.ts` re-applies every `.sql` file in lexicographic order on each app start. This means every statement must be safe to run twice: use `CREATE TABLE IF NOT EXISTS`, `CREATE OR REPLACE VIEW`, and `INSERT … ON DUPLICATE KEY UPDATE id = id` for seed rows. Never use `DROP`, `DELETE`, `TRUNCATE`, or any `ALTER TABLE` that would break on a second execution. The idempotency rule is a hard guardrail, not a suggestion.

- **When a MySQL VIEW reads a settings row on every query, use a subquery not a JOIN.** `vw_archive_coverage_target` reads `(SELECT play_lead_days FROM app_archive_settings WHERE id=1)` inline in a CTE. This is acceptable at ~92 tickers because MySQL evaluates the subquery once per query execution, not once per row. If the view ever slows down under larger data, materialise the settings values in the Python script instead and remove the subquery from the view.

- **TA indicator maths must match pandas, not the textbook formulae.** `src/modules/ta-charts/indicators.ts` is ported from the legacy `technical_analysis_batch.py` and is unit-tested to 6 dp against a golden xlsx fixture (`indicators.test.ts`). Three traps where the obvious/textbook version diverges and silently corrupts every downstream number: (1) **EMA/MACD** is pandas `ewm(span, adjust=False)` — recursive, **seeded at the first value** (`out[0]=x[0]`), `α=2/(span+1)`; it is *not* an SMA-seeded EMA. (2) **Stochastic** uses the **High/Low** channel (`min(Low,k)`/`max(High,k)`), **not** Close for the highs/lows. (3) **RSI** is pandas `ewm(alpha=1/period, adjust=False)` over gains/losses with the first delta treated as 0 and the averages **seeded at 0** — *not* the classic Wilder simple-average-of-first-N seed, and not a plain rolling mean. If you ever re-derive or "optimise" these, re-run `npx vitest run` — the golden test is the guard.

- **TA `detectSignals` must dedupe per bar (last-writer-wins) to match the legacy.** The legacy `detect_signals()` writes a boolean column (`data.loc[signal_date,'Buy_Signal']=True`), so when two consecutive Stochastic crosses both resolve to the *same* MACD-cross day, that bar is flagged once — and because the loop runs `i` ascending and overwrites the cross-date/days-between, the **last (largest) stoch cross wins**. A naive array that `push`es one signal per qualifying `i` produces duplicate markers at the same bar and fails parity. Fix: collect into a `Map` keyed by the MACD-cross bar index and `set()` in `i` order (later overwrites earlier). The golden test (`detectSignals` vs the xlsx Buy/Sell columns) catches this.

- **No test runner existed before TA Charts — Vitest was added (2026-05-30).** `npm run test` (watch) / `npm run test:run` (CI). Config: `vitest.config.ts` (root, `environment: 'node'`, `include: ['src/**/*.test.ts']`). Pure-logic suites only for now (`indicators.test.ts`). The JSON fixture lives in `src/modules/ta-charts/__fixtures__/` and imports fine because the base tsconfig already enables `resolveJsonModule`. `electron-vite build` does not bundle test/fixture files (they're unreachable from the app entry).

- **New IPC namespace for archive CSV import: `archive:previewPriceCsv` / `archive:importPriceCsv`.** These live in `electron/ipc/archive.ts` (registered via `registerArchiveHandlers()` in `main.ts`). The handler needs a transactional connection, so it calls `getPool()` exported from `electron/ipc/db.ts` (the pool was previously module-private). When adding handlers that need transactions, export `getPool()` rather than duplicating the mysql config.

- **`IF NOT EXISTS` is NOT valid for `ALTER TABLE … ADD COLUMN` / `ADD INDEX` in MySQL 8.0.** It is a MariaDB extension; on real MySQL 8.0 both throw `ER_PARSE_ERROR` (1064) — and since `runMigrations()` runs the whole `.sql` via one `conn.query`, a parse error aborts the *entire* migration run, so the DB never initialises and every later `db:query` fails with "Database not initialised". (Confirmed the hard way on migration 008.) To add a column idempotently, guard with `information_schema` + a prepared statement (the runner uses `multipleStatements: true`, so a `;`-separated block runs as one query):
  ```sql
  SET @col_exists := (SELECT COUNT(*) FROM information_schema.COLUMNS
    WHERE TABLE_SCHEMA = DATABASE() AND TABLE_NAME = 't' AND COLUMN_NAME = 'c');
  SET @ddl := IF(@col_exists = 0, 'ALTER TABLE t ADD COLUMN c …', 'DO 0');
  PREPARE stmt FROM @ddl; EXECUTE stmt; DEALLOCATE PREPARE stmt;
  ```
  Same pattern (against `information_schema.STATISTICS`) for `ADD INDEX`. `CREATE TABLE IF NOT EXISTS` and `CREATE OR REPLACE VIEW` *are* valid and remain the preferred idempotent forms.

- **A Web Worker in the renderer cannot use `DedicatedWorkerGlobalScope` / `self.onmessage` types — `tsconfig.web.json` targets DOM, not the `WebWorker` lib.** Referencing `DedicatedWorkerGlobalScope` fails `tsc` with `TS2304: Cannot find name`. Don't add the WebWorker lib (it conflicts with DOM's `self: Window`). Instead cast `self` to a minimal local interface: `const ctx = self as unknown as { onmessage: ((e: MessageEvent<In>) => void) | null; postMessage: (m: unknown) => void }`. The worker file (`aggregate.worker.ts`) is instantiated with `new Worker(new URL('./x.worker.ts', import.meta.url), { type: 'module' })` — Vite bundles it as its own chunk automatically (confirmed: `assets/aggregate.worker-*.js`). IPC (`window.electronAPI`) is NOT available inside the worker, so fetch on the renderer thread and pass data in via `postMessage` (Map/objects are structured-cloneable). Keep the actual maths in a pure module both the worker and an inline fallback import (`runCohort.ts`), so `worker.onerror` can fall back to computing on the main thread.

- **PowerShell `Out-File -Encoding utf8` writes UTF-8 BOM + CRLF; bash redirection writes plain UTF-8 LF.** When comparing snapshot files written by PowerShell with files written by bash (or comparing across sessions), a byte-level `diff` will show all lines different even when the values are identical. Normalise before diffing: `tr -d '\r' < file | sed 's/^\xef\xbb\xbf//'`. Prefer running snapshot comparisons entirely in MySQL (load both into temp tables and diff there) or write both snapshots from the same tool.

- **`momentum_new` blanks come from the gate, NOT a NULL formula; and a blank coerces to 0 → a phantom pass in Excel `SUMPRODUCT`.** `calculate_priority_13_for_company` builds `momentum_new` as a sum of 19 trend rungs, each `COALESCE`-style wrapped so a missing operand contributes 0 — the formula never returns NULL. The value is blank only because the INSERT is gated `WHERE coverage_new.value >= N`. In Excel, a blank momentum_new cell is read as 0 by `SUMPRODUCT`, and because the matrix `momentum_new` thresholds are ≤ 0, the blank silently *passes* the criterion (phantom pass) — inflating play/play_2 counts vs the DB, which (correctly) fails a blank criterion. **2026-06-05:** gate lowered `0.5 → 0.4` (the only lever that un-blanks; switching rungs to 2y averages changes the score but NOT the gate, since `coverage_new` is a separate metric). Effect: full play=15 55→58, play_2=16 70→71 (the predicted +4/+2 from un-blanking ASO/IESC/SMCI/YELP and ASO/BLD, **minus 1 each** from a *correct* staleness fix on SUP.L 2024A whose stored momentum was stale-high). **Recompute order after any priority-13 change:** loop `calculate_priority_13_for_company(id)` over all `dim_companies` (proc self-cleans via DELETE, so idempotent) → `calculate_all_play_rankings()` → loop `calculate_sector_play_for_company(id)` → `sp_refresh_stock_archive_flat()`. Re-running priority 13 universe-wide also refreshes stale stored momentum values, which can independently shift a borderline play — expect small deviations from a pure-gate simulation.


- **`stock_archive_flat` is a materialised BASE TABLE, not a view — every proc that writes a metric it surfaces must run BEFORE the final `sp_refresh_stock_archive_flat`.** (Found 2026-06-17 in `populate_matrices_2.py`.) The script ran `sp_refresh_stock_archive_flat` immediately after `calculate_all_play_rankings`, but `calculate_all_sector_play_ratings` ran *later* and the flat table was never rebuilt again — so `play_sector_rating` / `play_2_sector_rating` in the flat table (and therefore the Master Excel that reads it) showed the **previous** run's values. Symptom: a few near-miss rows (play=14 / play_2=15) showed NULL where `fact_metrics` already held a correct 0/1, plus stale 0/1 clinging to full plays (play_2=16) whose tier had just changed. The calc procs were 100% correct (`fact_metrics` 153/153 and 210/210); only the materialised mirror was stale. Fix: compute everything first (play rankings → sector ratings), then call `sp_refresh_stock_archive_flat` **exactly once, last**, guarded by a `needs_recompute` flag; and recompute sector ratings whenever play scores changed OR the sector sheet changed (play-tier membership decides who is a near-miss), not only on a sector-sheet edit. **What NULL means here:** only the single near-miss tier ever gets a 0/1 — full plays (play=15 / play_2=16) and every lower score are NULL **by design**; don't "fix" those. Reconcile: `SELECT COUNT(*) FROM stock_archive_flat WHERE play=14 AND play_sector_rating IS NULL` must be 0 (same for play_2=15). Ties into the momentum_new recompute-order lesson below.

- **The same "refresh the flat table last" omission also existed in the *app's* Sector Signals apply path — not just the Python pipeline.** (Found 2026-06-19.) `handleApply` in `src/modules/sector-signals/index.tsx` UPSERTed the toggle into `dim_sector_play_*_matrix` then `CALL calculate_all_sector_play_ratings()` (correctly rewriting `fact_metrics`) but **never refreshed `stock_archive_flat`**. The app's Watching dashboard reads `view_watching` (live over `fact_metrics`) so it looked correct, while the Master Excel reads the materialised `stock_archive_flat` and stayed stale — toggling Industrials × Opportunity flipped BLBD's `play_2_sector_rating` to 1 in `fact_metrics`/app but left it 0 in the flat table/Excel. Fix: add `await window.electronAPI.db.callProc('sp_refresh_stock_archive_flat')` immediately after the recompute, last. **Rule:** any in-app action that recomputes a `fact_metrics` metric surfaced by `stock_archive_flat` must finish with the full `sp_refresh_stock_archive_flat()` (the recompute here is universe-wide, so the per-company `sp_refresh_flat_for_company` is insufficient). This is the only such app write path — every other module writes app-owned tables not surfaced by the flat table (`app_portfolio`, `app_date_overrides`, `app_ta_settings`, prices), so they need no refresh.

- **The earnings scanner's Q1/Q3 skip only ever covered Finnhub — FMP has no quarter, so its off-cycle dates silently drove the Report Calendar wrong.** (2026-07-07.) `earnings_scanner.py` skipped `record.quarter in (1, 3)`, but only Finnhub populates `quarter`; `FMPClient` hardcodes `quarter=None`, so no FMP row was ever filtered. FMP surfaces real US quarterlies (TNK's Q1, 13 May) **and fabricates a Q1 for UK semiannual reporters** (TIG.L's "15 June" carries `fiscalDateEnding=2026-03-31` — FMP applies its quarterly frame to a company that files none). Fix at source: FMP's `/v3/earning_calendar` returns `fiscalDateEnding` on 100% of rows (it was being discarded); drop an FMP row when that period end isn't within **±5 days** of the ticker's real annual/semiannual period ends read from `fact_reports.report_date` (year-wrap aware). Keep it when `fiscalDateEnding` is missing/unparseable and let the view guards catch it. **Two systemic reminders:** (1) when two data sources feed one field, check each one *actually carries* the label you filter on — a filter that no-ops on half the sources looks like it works; (2) filter at the ticker's *own* historical cadence, not a hardcoded calendar, so any fiscal-year-end is handled.

- **`view_earnings_calendar` picks the API date CLOSEST to its own estimate, not the earliest — and the "trust the estimate" guard is 45 days, inlined in THREE places.** (Phase-6, `docs/decisions/play-engine/phase-6-calendar-api-match.sql`.) The old `best_api_date = min(finnhub, fmp)` grabbed FMP's Q1 (May) over Finnhub's correct Q2 (Jul); now it's `argmin |api - estimated_release_date|` (tie → Finnhub). This required moving the `estimated_release_date` CTE *before* the API-selection CTE (`calendar_ref → calendar_est → calendar_api`), because you cannot reference a SELECT-list alias from another expression in the same SELECT — the estimate must be computed in an earlier CTE. The 45-day guard (`estimated - best_api >= 45 → use estimate`) is duplicated in `relevant_date`, `date_source`, **and** the `is_already_reviewed` EXISTS subquery — change all three together or they diverge.

- **Re-creating a view under the wrong `collation_connection` silently breaks any view that compares its string columns to a literal.** (2026-07-07.) `view_earnings_calendar`'s `calendar_status`/`date_source`/`urgency` are CASE-of-string-literals, so their collation is whatever `collation_connection` was at `CREATE` time. mysql2's default session made them `utf8mb4_unicode_ci`, but the schema default is `utf8mb4_0900_ai_ci` and `view_earnings_calendar_actionable`'s stored `WHERE calendar_status = 'actionable'` literal is `0900_ai_ci` — so every query against `_actionable` then threw `ER_CANT_AGGREGATE_2COLLATIONS (1267)`. The dependent view compiles fine; it only fails at query time. Fix: pin `SET NAMES utf8mb4 COLLATE utf8mb4_0900_ai_ci;` at the top of the view script so it's reproducible from any client. When replacing a view whose string columns are consumed by another object, verify the output-column collations (`information_schema.COLUMNS`) match the original, and smoke-query every dependent view.

- **Playwright `setViewportSize()` silently fails on Electron windows — smoke screenshots crop the window and fake layout bugs.** The Stage-6 Dashboards smoke called `win.setViewportSize({1500×950}).catch(() => {})`; the call no-opped (the real window was 1643 px wide), so `win.screenshot()` captured a 1500-px crop whose missing right edge looked exactly like the STATUS column being clipped by `overflow: hidden`. An in-page measurement (`wrap.scrollWidth === wrap.clientWidth === table.offsetWidth`) proved there was no overflow at all. Rule: before "fixing" a layout problem seen only in a smoke screenshot, measure it in-page with `win.evaluate()` (scrollWidth vs clientWidth, element offsets) — a screenshot of an Electron window is not evidence of clipping unless the captured width matches `window.innerWidth`.
